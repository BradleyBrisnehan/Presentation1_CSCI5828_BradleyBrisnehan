Weaknesses

Rapid Application Development has several weaknesses:

  • Users expect prototypes to be readily converted to the final system. The user/customer may become impatient during the development and integration time.
  • System complexity is easily expanded. No method exists to limit the difficulty and extent of functionality changes requested by the user/customer.
  • Time spent building throwaway prototypes is lost. Developers may try to force integration of other types of prototypes into the final design - when rebuilding is required.

These weaknesses make Rapid Application Development a poor choice for certain types of projects:

  • Poorly Defined Scope If the scope is not very well defined or established, RAD may exacerbate the desire to continually add functionality and create a never-ending project.
  • Large Technological Development If a large technical advancement is required as part of the project, it may not receive the attention needed under the prototype model. RAD is better suited for projects in which the technology is already in place and well-established.
  • Large Data Sets If a volumous amount of data must be analyzed as part of the project, it may not receive the attention needed under the prototype model. RAD is better suited for projects in which data sets have been previously analyzed and minimal data analysis is required.
